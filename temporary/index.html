TR64 finish-line — one PowerShell line (v2)

⚠ Paste this in Windows PowerShell (Administrator). NOT in an SSH session to the mini — that's bash, this is PowerShell syntax.

Start-Service sshd; Set-Service sshd -StartupType Automatic; if(-not(Get-NetFirewallRule -Name OpenSSH-Server-In-TCP -EA SilentlyContinue)){New-NetFirewallRule -Name OpenSSH-Server-In-TCP -DisplayName "OpenSSH Server" -Enabled True -Direction Inbound -Protocol TCP -Action Allow -LocalPort 22|Out-Null}; New-Item -ItemType Directory -Path C:\ProgramData\ssh -Force|Out-Null; Set-Content C:\ProgramData\ssh\administrators_authorized_keys 'ssh-ed25519 AAAAC3NzaC1lZDI1NTE5AAAAIH79Y9ygZLTFENFsN4sIx+BpFjbRJs7DX5/qJhnHZ+xC nebula@nebula-controller'; icacls C:\ProgramData\ssh\administrators_authorized_keys /inheritance:r /grant "Administrators:F" "SYSTEM:F"|Out-Null; Get-NetTCPConnection -State Listen -LocalPort 22

Uses C:\ProgramData\ssh\administrators_authorized_keys — Windows OpenSSH's canonical place for keys that apply to any administrator account. Skips the need for a separate nebula Windows user (which didn't exist on this box and made v1 fail with PermissionDenied: C:\Users\nebula\.ssh).

What it does (7 chained statements):

  1. Start sshd + set automatic startup
  2. Open firewall TCP :22 inbound (if not already)
  3. Create C:\ProgramData\ssh
  4. Drop Nebula-HQ pub key into administrators_authorized_keys
  5. Lock down ACL to Administrators + SYSTEM only
  6. Print the listening row → proves :22 is up

Expected last line of output: a table row showing LocalAddress=0.0.0.0 LocalPort=22 State=Listen.

After this, mini → Windows SSH works as ssh Admin@10.3.1.88 (not nebula@ — there's no nebula user on the Windows box; the key is authorized for any administrator).
